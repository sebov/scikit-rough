
SUMMARY OF DISCUSSION: HASH-BASED GROUPING, HISTOGRAMS, AND PERFORMANCE TRADE-OFFS

1. CORE PROBLEM
User designs an iterative system where:
- A vector X is transformed column-by-column into a rolling hash:
  hash = hash * B + C
- This creates a compact representation of multi-column vectors.
- The hash defines group_id (bucket assignment).
- For each group, user computes a distribution over decision values.

2. MAIN DATA STRUCTURE
- group_id per row (from hash)
- decision_id per row (separate array)
- histogram Z[group_id, decision_id] += 1

This yields:
- Z = contingency table / P(decision | group)

3. KEY DESIGN CONSTRAINTS
- Many iterations over different C configurations
- Z is temporary (computed, scored, discarded)
- Need high performance (Numba/Rust consideration)
- Memory and cache efficiency are critical

4. GROUP ID COMPACTION PROBLEM
Issue:
- hash space is sparse and potentially large
Options discussed:

A) Dense compaction (recommended baseline)
- map group_id → compact_id (0..K-1)
- build Z as dense array

Pros:
- fast indexing
- cache-friendly
- minimal memory

Cons:
- compaction cost

B) Hashmap-based dynamic indexing (M + Z')
- group_id → compact_id via dict during runtime

Pros:
- no preprocessing
Cons:
- slow (dict overhead, branches, cache misses)

C) Sort-based grouping (rejected for this case)
- sort by group_id, then scan groups

Pros:
- no hashmap
Cons:
- O(n log n)
- higher memory traffic
- not suitable for streaming/iterative loop

5. BEST PRACTICE CONCLUSION
Best approach:
- Dense arrays + compaction per iteration
- Reuse Z buffer between runs (Z.fill(0))
- Avoid hashmaps in hot loop
- Avoid sorting

6. PERFORMANCE INSIGHT (CACHE STABILITY)
- CPU prefers contiguous memory access
- Reusing Z improves cache locality
- Allocation and pointer chasing destroy cache efficiency
- Stable memory layout is critical for performance

7. NUMBA VS RUST DISCUSSION

NUMBA:
- Best fit for this workload
- Compiles Python loops to near C performance
- Ideal for array-based histogram updates
- Minimal overhead

RUST (maturin extension):
- Best for production-grade kernels
- Full memory control, SIMD potential
- Higher complexity and development cost
- Only needed if Numba is insufficient

Conclusion:
- Start with Numba
- Move to Rust only if profiling shows bottlenecks beyond Numba limits

8. MEMORY MANAGEMENT STRATEGY

Recommended pattern:
- Allocate Z in Python once
- Pass Z into Numba function
- Modify in-place
- Reset via Z.fill(0)
- Optionally grow buffer if needed

9. FINAL ARCHITECTURE

For each iteration C:
1. Compute group_id (hash-based)
2. Compact group IDs (optional but recommended)
3. Update histogram:
   Z[group_id, decision_id] += 1
4. Compute score per group
5. Aggregate score
6. Reset Z for next iteration

10. KEY TAKEAWAYS

- Dense arrays outperform hashmaps in this workload
- Sorting is not beneficial for this streaming-style pipeline
- Main optimization is cache efficiency, not asymptotic complexity
- Numba provides near-optimal performance for this design
- Rust is only a later-stage optimization if needed
